# 架构优化建议（待确认，不在本次重构）

本清单基于当前单机、单用户、Ubuntu + Docker Compose 的产品定位。建议继续采用模块化单体，不建议为了“看起来先进”拆成大量微服务。

## P0：建议优先确认

### 1. 清理并模块化 CSS

现状：`src/styles.css` 约 3488 行，同一工作区布局和媒体查询经过多轮覆盖。本次“预览覆盖统计栏”的根因就是较早的响应式规则被文件尾部规则覆盖。

建议：

- 按 `navigation / workspace / import-dialog / platform / responsive` 拆分样式文件。
- 每个组件只保留一处布局定义；响应式规则紧邻组件样式。
- 引入明确断点变量，并持续保留“不重叠、无横向滚动”的浏览器断言。

收益：显著降低 UI 回归概率，是当前投资回报最高的重构。

### 2. 拆分前端巨型组件

现状：`src/main.jsx` 约 2641 行，项目管理、导入、训练、推理、评估及弹窗状态集中在一个文件，包含约 44 处直接 `fetch`。

建议：

- 按功能拆为 `features/projects`、`features/imports`、`features/training`、`features/inference`。
- 抽出统一 API client、错误处理和轮询 hooks。
- 将工作区状态放入专用 `useWorkspace`，避免页面级状态互相影响。

收益：功能修改范围更明确，测试更容易，减少状态联动 bug。

### 3. 拆分后端模块化边界

现状：`server/postgres-app.js` 约 3736 行，同时承担 HTTP 路由、导入扫描、项目管理、导出、训练调度、推理和系统对话框代理。

建议：

- 路由层只负责参数和响应。
- 业务拆为 project/import/export/model/runtime services。
- SQL 收敛到 repository 层。
- 后台任务保留同一镜像，但用独立 worker 进程运行，避免耗时导入或训练影响 Web API。

收益：保持单体部署的简洁，同时获得清晰边界和故障隔离。

## P1：随后处理

### 4. 统一数据库迁移

现状：`db/schema.sql` 与后端启动时约 59 条 `CREATE/ALTER` 语句共同维护结构，存在双源漂移风险。

建议：采用按版本编号的 migrations；全新安装和升级都走同一迁移链。

### 5. 合并两套 Compose 和启动逻辑

现状：源码便携模式与离线发布分别维护 `docker-compose.portable.yml`、`deploy/compose.yml` 及不同启动脚本。

建议：以一份基础 Compose 为主，通过 override/profile 区分源码构建、离线镜像和 GPU。

### 6. 强化原生选择器生命周期

本次已增加令牌、后端代理、启动连通检查和自动停止。长期建议将桥接安装为 systemd user service，获得崩溃重启和统一日志；当前脚本模式仍适合免安装压缩包。

### 7. 完善多文件导入规划器

本次已支持文件夹、单文件和多文件。单独选择图片时会自动读取同目录标注文件；复杂的跨目录 COCO/YOLO 标注最好增加“导入预览”，在正式写入前展示将匹配的图片、标注和警告。

## P2：发布成熟度

### 8. 增加备份策略

- 备份保留周期和自动清理。
- 可选加密与异机复制。
- CI 定期执行真实恢复演练。

### 9. 增加视觉与可访问性测试

- 关键宽度截图回归。
- 125%、150%、200% 浏览器缩放测试。
- 键盘导航、焦点和 aria 检查。

### 10. 明确局域网安全边界

当前无账号体系符合单用户要求，默认仍应只监听 `127.0.0.1`。若设置为 `0.0.0.0` 提供局域网访问，建议后续至少增加反向代理访问口令或可信网段限制。

## 推荐实施顺序

1. CSS 模块化。
2. 前端工作区与导入模块拆分。
3. 后端 route/service/repository 拆分。
4. 统一迁移与 Compose。
5. 独立 worker、备份策略和视觉测试。

请确认优先级后再开始这些结构性重构；本次功能修复不包含上述大规模改造。
