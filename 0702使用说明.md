## 1. 先看结论

0702 版本已经从“主要支持逐图片 LabelMe 的本地原型”，整理为可以通过 Docker Compose 部署的单机数据集管理程序。

本次最重要的变化是：

1. 修复 Ubuntu 文件夹浏览和“上一级”导航问题。
2. 浏览器可以查看配置范围内的真实 Ubuntu 文件系统，默认浏览根为 `/`。
3. 场景筛选不再只显示 `UnknownScene`，系统可以从标注内容或上级目录自动识别场景、日期等属性。
4. 导入从仅支持逐图片 LabelMe，扩展到 LabelMe、标准 COCO、YOLO 检测和 YOLO 分割行。
5. 导出增加 LabelMe、标准 COCO、YOLO 三种格式。
6. 增加单元测试、真实 PostgreSQL/MinIO API 测试和 Chrome 浏览器端到端测试。
7. Docker 镜像改为非 root 用户、只读根文件系统、多阶段构建，并在构建镜像时执行核心测试。
8. README、架构说明、审查报告和 CI 流程已统一到当前 Ubuntu + Docker 方案。

推荐新人直接使用 Docker 方式，不需要先理解 React、Node.js、PostgreSQL 或 MinIO。

## 2. 与以前版本的详细对比

| 项目 | 以前 | 0702 版本 |
| --- | --- | --- |
| 文件夹选择 | 主要依赖网页内目录列表，行为容易与 Ubuntu 本机路径不一致 | 优先调用 Ubuntu `zenity` 原生文件夹选择器；不可用时自动回退网页选择器 |
| 浏览范围 | 容器路径和宿主路径容易混淆 | 使用内部路径与显示路径双重映射，页面显示真实 Ubuntu 路径 |
| “上一级” | 特定路径配置下点击无响应或接口返回 500 | 按浏览作用域独立映射，并限制在安全根目录内导航 |
| 文件系统安全 | 路径边界不够集中 | 浏览挂载只读，路径映射后再次校验，路径穿越请求返回 403 |
| 场景属性 | 缺少 `scene` 时经常得到 `UnknownScene` | 优先读标注明示属性，否则从最近的语义上级目录自动推断 |
| LabelMe 导入 | 支持逐图片 JSON，但匹配和异常处理能力有限 | 支持相对路径、同名文件、相邻目录匹配，非法框会被拒绝 |
| COCO 导入 | 标准 COCO 单 JSON 不支持 | 支持 `images/categories/annotations` 和 `bbox` |
| YOLO 导入 | 不支持 | 支持检测框与 polygon 行，类别名读取 YAML 或 `.names` |
| 导出格式 | 只有 LabelMe | 可在页面选择 LabelMe、COCO 或 YOLO |
| 图片尺寸 | 标注未写尺寸时可能影响归一化坐标 | 使用 Sharp 读取真实图片尺寸并回填 |
| Linux 文件名 | 路径无条件转小写可能混淆大小写不同的文件 | Windows 才忽略大小写，Ubuntu 保留真实大小写 |
| 无效检测框 | 零面积或异常框可能进入后续流程 | 非有限值、负尺寸、零面积框会被拒绝 |
| 导出查询 | 存在逐图片读取标注的 N+1 查询 | 一次批量加载当前版本标注，再按图片分组 |
| Docker 构建 | 构建镜像不运行核心测试 | Dockerfile 增加测试阶段，测试失败则不能生成运行镜像 |
| Docker 权限 | 容易留下 root 创建的数据目录 | 使用一次性 permissions 服务修复目录属主，应用使用本机 UID/GID |
| 容器权限 | 运行边界较宽 | 非 root、只读根文件系统、移除 capabilities、禁止提权 |
| 自动化测试 | 覆盖不足 | 单元测试、API 全流程、Playwright 浏览器流程、隔离 Docker 全栈 |
| CI | 以构建检查为主 | 单元测试 → 集成测试 → 多架构镜像构建，并使用当前 Actions 运行时 |
| 文档 | 存在旧 Windows 路径和过时能力说明 | 统一为 Ubuntu 单机 Docker 发布方式 |

## 3. 当前已经支持什么

### 3.1 项目和数据管理

- 新建普通项目。
- 查看项目中的图片、视频和标注数量。
- 删除项目到回收站。
- 从回收站恢复项目。
- 管理导入批次，支持删除和恢复。
- 图片和视频按 SHA-256 去重，重复文件不会重复保存完整对象。

### 3.2 数据浏览与筛选

- 缩略图分页浏览。
- 打开图片详情。
- 图片缩放和平移。
- 按场景、视角、模态、标注类别和导入批次筛选。
- 从文件夹名称自动生成场景属性。
- 从 LabelMe 或 COCO 中显式填写的 `scene`、`view` 等字段读取属性。

### 3.3 标注

- 在网页中绘制矩形检测框。
- 移动、缩放、删除检测框。
- 修改类别并保存当前图片标注。
- 导入 LabelMe、COCO、YOLO 后统一转换到内部矩形框模型。

### 3.4 导出

- LabelMe：每张图片一个 JSON。
- COCO：单个 `instances.json`。
- YOLO：每张图片一个 TXT，并生成 `data.yaml`。

### 3.5 模型管理

- 管理模型、模型版本、训练模板。
- 创建训练和推理任务记录。
- 查看任务状态和日志记录。

注意：便携 Docker 配置默认关闭训练 worker。0702 测试只验证任务创建、查询和重新排队，没有实际执行训练或推理。

## 4. 当前还没有完整支持什么

以下内容不要误认为已经完成：

- 视频可以导入、去重、统计、删除和恢复，但还没有视频播放器、抽帧、时间轴和轨迹标注工作台。
- 内部标注模型目前是矩形框。COCO segmentation 或 YOLO polygon 导入后使用外接矩形，不能无损还原原始多边形。
- 没有账号登录、用户权限、租户隔离和 TLS，不适合直接暴露到公网。
- 数据库尚未采用正式版本化 migration 工具。
- 后台任务仍运行在 API 进程中，不是独立队列 worker。
- 跨电脑恢复、备份恢复和 ARM 实体机运行没有在本次任务中实际验证。

## 5. 新人第一次安装：推荐 Docker 方法

下面的命令默认在 Ubuntu 22.04 或更新版本执行。

### 5.1 检查 Git

打开终端，执行：

```bash
git --version
```

如果提示找不到命令：

```bash
sudo apt update
sudo apt install -y git
```

### 5.2 安装 Docker

如果已经提前下载了本仓库，可以运行仓库提供的安装脚本：

```bash
bash scripts/install-docker-ubuntu.sh
```

如果此时还没有克隆仓库，请先按照下一节完成克隆，再返回执行这个脚本；也可以直接按照 Docker 官方 Ubuntu 安装文档安装 Docker Engine 和 Compose Plugin。

安装后检查：

```bash
docker --version
docker compose version
docker info
```

如果 `docker info` 出现 socket 权限错误：

```bash
sudo usermod -aG docker "$USER"
newgrp docker
docker info
```

如果仍然没有权限，注销 Ubuntu 当前用户并重新登录，再执行 `docker info`。

### 5.3 克隆正确仓库和分支

```bash
git clone https://github.com/GodBarry/Det-DashBoard.git
cd Det-DashBoard
git switch ZBH
```

确认当前分支：

```bash
git branch --show-current
```

输出应为：

```text
ZBH
```

### 5.4 创建本机配置

```bash
cp .env.portable.example .env.portable
```

用文本编辑器打开：

```bash
nano .env.portable
```

至少修改这两个密码：

```dotenv
POSTGRES_PASSWORD=换成自己的强密码
MINIO_ROOT_PASSWORD=换成自己的强密码
```

在 nano 中按 `Ctrl+O` 保存，回车确认，再按 `Ctrl+X` 退出。

`.env.portable` 已被 Git 忽略，不应提交到 GitHub。

### 5.5 启动程序

```bash
bash scripts/portable-start.sh
```

第一次启动需要下载基础镜像和安装 Node 依赖，时间取决于网络速度。脚本会：

1. 创建持久化目录。
2. 构建应用镜像。
3. 在镜像构建期间运行单元测试。
4. 启动 PostgreSQL、MinIO 和应用。
5. 修复历史数据目录权限。
6. 等待服务健康。
7. 检测 Ubuntu 原生文件夹选择器是否可用。

看到下面内容表示启动成功：

```text
Det-DashBoard is running at http://localhost:5173
```

打开浏览器：

```text
http://localhost:5173
```

### 5.6 检查运行状态

```bash
docker compose --env-file .env.portable -f docker-compose.portable.yml ps
```

预期状态：

- `app`：`healthy`
- `postgres`：`healthy`
- `minio`：`healthy`
- `permissions`：退出码 `0`，这是正常情况，因为它只执行一次权限初始化

检查应用 API：

```bash
curl -fsS http://localhost:5173/api/health/live
curl -fsS http://localhost:5173/api/health/ready
```

正常返回：

```json
{"status":"ok"}
```

### 5.7 停止和再次启动

停止：

```bash
bash scripts/portable-stop.sh
```

再次启动：

```bash
bash scripts/portable-start.sh
```

普通停止不会删除项目、数据库、MinIO 对象或导出结果。

## 6. 文件和数据到底保存在哪里

仓库运行后主要目录如下：

```text
Det-DashBoard/
├── .env.portable              # 本机配置和密码，不提交 Git
├── datasets/                  # 默认数据集挂载目录
├── exports/                   # 导出结果
└── portable-data/
    ├── postgres/              # PostgreSQL 数据库文件
    ├── minio/                 # 导入后的图片、视频和原始标注对象
    ├── storage/               # 缩略图、临时文件、可选 fallback
    ├── folder-dialog.log      # 原生文件夹选择器日志
    └── folder-dialog.pid      # 原生选择桥接进程号
```

不要手动修改 `portable-data/postgres` 和 `portable-data/minio` 内部文件。

默认 `OBJECT_STORE_WRITE_FALLBACK=false`，导入文件会复制进 MinIO。此时即使原始图片后来被移动，已导入数据通常仍可使用。

如果改成：

```dotenv
OBJECT_STORE_WRITE_FALLBACK=true
```

系统可能使用链接或本地 fallback 来降低重复磁盘占用，但移动或删除原文件可能导致数据不可读。新人建议保持 `false`。

## 7. 为什么网页可以浏览 Ubuntu 文件夹

Docker 容器默认无法直接看到宿主机文件。便携配置把指定宿主目录只读挂载到容器：

```dotenv
HOST_BROWSE_ROOT=/
BROWSE_ROOT_DISPLAY=/
```

这表示：

- 容器可以只读浏览整个 Ubuntu 文件系统。
- 页面显示 `/home/用户名/...` 等真实宿主路径。
- 应用不能通过这个挂载修改或删除原始文件。
- 应用只能写自己的 storage 和 exports 目录。

如果不想让程序浏览整个系统，可以收窄范围：

```dotenv
HOST_BROWSE_ROOT=/home/barry/图片
BROWSE_ROOT_DISPLAY=/home/barry/图片
```

修改 `.env.portable` 后重新启动：

```bash
bash scripts/portable-stop.sh
bash scripts/portable-start.sh
```

网页的“上一级”只能返回到配置的浏览根。如果浏览根是 `/home/barry/图片`，到达这个目录后按钮不能继续返回 `/home/barry`，这是安全边界，不是故障。

## 8. Ubuntu 原生文件夹选择器

系统优先尝试调用 Ubuntu 的 `zenity`。需要满足：

- 宿主机安装 Node.js。
- 宿主机安装 `zenity`。
- 当前是图形桌面会话。
- `DISPLAY` 或 `WAYLAND_DISPLAY` 可用。

检查：

```bash
node --version
command -v zenity
echo "$DISPLAY"
echo "$WAYLAND_DISPLAY"
```

安装 zenity：

```bash
sudo apt update
sudo apt install -y zenity
```

如果原生窗口没有弹出，查看：

```bash
cat portable-data/folder-dialog.log
```

原生选择器不可用时，程序会自动使用网页文件夹浏览器，不会阻止导入。

## 9. 第一次使用网页

### 9.1 新建项目

1. 打开 `http://localhost:5173`。
2. 点击“新建项目”。
3. 输入项目名称，例如 `2026年7月山地数据`。
4. 确认创建。
5. 点击项目卡片进入工作台。

建议一个项目对应一个明确的数据用途，不要把完全无关的数据全部放进同一项目。

### 9.2 导入数据

1. 进入项目。
2. 点击“导入数据”。
3. 点击“浏览”。
4. 在原生选择器或网页选择器中进入数据集目录。
5. 选择数据集的共同上级目录，而不是只选择某一张图片。
6. 点击“开始导入”。
7. 等待状态从扫描、运行变成完成。

同一项目同一时间只允许一个导入任务。导入期间不要关闭 Docker。

### 9.3 导入完成后检查

至少检查：

- 图片数是否符合预期。
- 标注数是否合理。
- 左侧场景是否出现真实目录名称，而不是全部 `UnknownScene`。
- 类别筛选中是否出现预期类别。
- 随机打开几张图片，框的位置是否正确。
- 导入批次信息中是否有警告。

## 10. 支持的数据格式

### 10.1 图片和视频扩展名

图片：

```text
.jpg .jpeg .png .bmp .webp
```

视频：

```text
.mp4 .avi .mov .mkv .wmv
```

### 10.2 LabelMe

推荐目录：

```text
山地/
├── images/
│   ├── 000001.jpg
│   └── 000002.jpg
└── jsons/
    ├── 000001.json
    └── 000002.json
```

示例 JSON：

```json
{
  "imagePath": "../images/000001.jpg",
  "imageHeight": 1080,
  "imageWidth": 1920,
  "view": "AerialView",
  "scene": "山地",
  "keyword": "",
  "shapes": [
    {
      "label": "vehicle",
      "points": [[100, 120], [300, 260]],
      "shape_type": "rectangle"
    }
  ]
}
```

图片匹配顺序包括：

1. `imagePath` 指向的相对路径。
2. JSON 同目录或上级目录。
3. 相邻 `images` 目录。
4. 同名图片和 JSON。

如果不同文件夹中存在完全相同的图片文件名，应保持清晰的相对路径，避免无法唯一匹配。

### 10.3 标准 COCO

推荐目录：

```text
港口/
├── images/
│   ├── 000001.jpg
│   └── 000002.jpg
└── annotations/
    └── instances.json
```

`instances.json` 至少应具有：

```json
{
  "images": [
    {"id": 1, "file_name": "000001.jpg", "width": 1920, "height": 1080}
  ],
  "categories": [
    {"id": 1, "name": "ship"}
  ],
  "annotations": [
    {"id": 1, "image_id": 1, "category_id": 1, "bbox": [100, 120, 200, 140]}
  ]
}
```

COCO 的 `bbox` 含义是：

```text
[左上角 x, 左上角 y, 宽度, 高度]
```

系统通过 `file_name` 匹配图片。若同名图片出现多次且路径信息无法唯一判断，系统会记录警告，不会随意匹配。

### 10.4 YOLO

推荐目录：

```text
草原/
├── data.yaml
├── images/
│   ├── train/
│   │   └── 000001.jpg
│   └── val/
│       └── 000002.jpg
└── labels/
    ├── train/
    │   └── 000001.txt
    └── val/
        └── 000002.txt
```

`data.yaml` 示例：

```yaml
names:
  0: person
  1: vehicle
```

检测框 TXT 行：

```text
1 0.500000 0.500000 0.200000 0.100000
```

字段含义：

```text
类别编号 中心点x 中心点y 宽度 高度
```

坐标均为相对图片宽高的归一化值。

系统也能读取 YOLO polygon 行，但会把多边形转成外接矩形框。类别名称优先从 `data.yaml`、`dataset.yaml` 或 `.names` 读取；找不到时显示为 `class_0`、`class_1` 等。

### 10.5 多种标注同时存在时

同一图片若同时存在多个格式，优先级是：

```text
LabelMe > COCO > YOLO
```

低优先级重复标注会记录为警告，避免同一图片重复叠加多个来源的框。

## 11. 场景属性如何自动识别

识别优先级：

1. LabelMe 或 COCO 中明确填写的 `scene`。
2. 从图片目录向上查找最近的语义目录。
3. 跳过结构性目录。

会被跳过的常见结构目录包括：

```text
images jsons annotations labels train val test
visible infrared 可见光 红外
```

示例：

```text
/数据/山地/2026-07-01/images/train/000001.jpg
```

系统从图片向上查找时会跳过 `train` 和 `images`，优先得到 `2026-07-01`。该值会显示在左侧“场景”筛选中。

如果场景仍是 `UnknownScene`：

- 检查标注中的 `scene` 是否为空。
- 检查图片上级是否只有 `images/train/val` 等结构目录。
- 尽量在图片目录外再增加一个场景或日期目录。
- 检查导入批次保存的来源路径是否仍可访问。

历史 `UnknownScene` 只会在目录结构无歧义时保守回填，避免自动写入错误属性。

## 12. 图片筛选与标注编辑

### 12.1 筛选

左侧筛选项包括：

- 场景。
- 视角。
- 模态。
- 类别。
- 导入批次。

选择多个条件后，只显示同时满足条件的图片。若结果为空，先逐项取消筛选，判断是哪一个条件不匹配。

### 12.2 编辑矩形框

1. 点击缩略图进入图片详情。
2. 选择或输入类别。
3. 在图片上拖动绘制矩形框。
4. 选择已有框后可以移动或调整尺寸。
5. 删除不需要的框。
6. 点击保存。

保存前建议确认图片缩放比例和框位置。系统内部保存的是原始图片像素坐标，不是浏览器显示尺寸。

## 13. 导出数据集

### 13.1 页面操作

1. 进入项目工作台。
2. 在顶部“导出格式”中选择 LabelMe、COCO 或 YOLO。
3. 点击“导出数据集”。
4. 等待后台任务完成。
5. 在宿主机 `exports` 目录查看结果。

导出根目录格式：

```text
exports/<项目名>_<时间戳>_<格式>/
```

### 13.2 LabelMe 导出结构

```text
项目_时间_labelme/
├── images/
└── jsons/
```

### 13.3 COCO 导出结构

```text
项目_时间_coco/
├── images/
└── annotations/
    └── instances.json
```

### 13.4 YOLO 导出结构

```text
项目_时间_yolo/
├── images/
├── labels/
└── data.yaml
```

如果项目包含从 polygon 转换来的矩形框，导出时输出的是矩形检测结果，不会重新生成原始 polygon。

## 18. 常见问题排查

### 18.1 页面打不开

依次执行：

```bash
docker ps
curl -v http://localhost:5173/api/health/ready
docker compose --env-file .env.portable -f docker-compose.portable.yml logs --tail 200 app
```

检查 5173 是否被其他程序占用：

```bash
ss -ltnp | grep 5173
```

可以在 `.env.portable` 修改：

```dotenv
APP_PORT=15173
```

之后访问 `http://localhost:15173`。

### 18.2 PostgreSQL 端口冲突

修改：

```dotenv
DB_PORT=55433
```

容器内部仍使用 5432，不需要修改代码的 `DATABASE_URL`。

### 18.3 MinIO 端口冲突

修改：

```dotenv
MINIO_HOST_PORT=59010
MINIO_CONSOLE_HOST_PORT=59011
```

### 18.4 原生文件夹窗口不弹出

```bash
command -v node
command -v zenity
echo "$DISPLAY"
echo "$WAYLAND_DISPLAY"
cat portable-data/folder-dialog.log
```

可以继续使用网页文件夹选择器。

### 18.5 “上一级”不可点击

先判断当前目录是否已经是 `BROWSE_ROOT_DISPLAY`。如果是，按钮被禁用是预期安全行为。如果不是，查看 app 日志和 `/api/config`：

```bash
curl -fsS http://localhost:5173/api/config
```

重点检查 `browseRootDisplay` 是否是预期路径。

### 18.6 场景是 UnknownScene

- 确认标注是否填写 `scene`。
- 确认图片上面存在场景或日期目录。
- 不要只导入一个没有语义名称的 `images` 目录；尽量选择它的上级场景目录。
- 查看导入批次警告。

### 18.7 图片显示 NoSuchKey

这表示数据库有记录，但对象存储没有对应文件。检查：

```bash
docker compose --env-file .env.portable -f docker-compose.portable.yml logs --tail 200 minio
du -sh portable-data/minio
du -sh portable-data/storage
```

不要随意删除 `portable-data/minio`。

### 18.8 导入后标注数是 0

- LabelMe：检查 JSON 中是否有 `shapes`，以及 `imagePath` 是否能匹配图片。
- COCO：检查是否同时存在 `images`、`categories`、`annotations`。
- YOLO：检查图片和 TXT 是否同名，目录是否为对应的 `images` 与 `labels`。
- YOLO：检查 TXT 是否至少包含类别和四个坐标值。
- 检查导入批次消息中的警告数量。

### 18.9 导出目录为空

导出是后台任务。先等待任务完成，再检查：

```bash
ls -la exports
docker compose --env-file .env.portable -f docker-compose.portable.yml logs --tail 200 app
```

确认 `EXPORTS_DIR` 对当前用户和容器 UID/GID 可写。

### 18.10 磁盘越来越大

默认会把导入对象保存到 MinIO，属于预期行为。先统计：

```bash
du -sh portable-data/minio portable-data/storage exports
```

不要直接删除内部对象。先清理不需要的项目、导入批次和导出目录，并确认数据已有备份。

## 19. 安全说明

当前版本是本机单用户程序，默认端口只绑定 `127.0.0.1`。

不要把以下配置改成 `0.0.0.0` 后直接暴露公网：

```dotenv
APP_BIND_ADDRESS
DB_BIND_ADDRESS
MINIO_BIND_ADDRESS
MINIO_CONSOLE_BIND_ADDRESS
```

如果未来必须远程访问，至少需要：

- 登录认证和权限控制。
- HTTPS/TLS。
- 带认证的反向代理。
- 收窄 `HOST_BROWSE_ROOT`。
- 防火墙规则。
- 独立低权限数据库和 MinIO 账户。
- 正式密钥管理和备份策略。

不要提交：

- `.env`
- `.env.portable`
- 密码、token、访问密钥
- `portable-data`
- 私有数据集和导出结果
